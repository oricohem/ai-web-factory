# External UI Design Stack Integration v1

Date: 2026-05-03

## Why this exists

AI Web Factory already has strong internal engines for strategy, architecture, launch systems, design research, and critique. Even so, future agents can still struggle with high-fidelity UI execution if they try to invent every component, interaction pattern, and visual grammar from scratch.

This integration layer exists to solve that gap:

- keep AI Web Factory's business and conversion thinking internal and original
- allow optional external UI/design stack support for execution quality
- improve visual outcomes without outsourcing product strategy

This does not guarantee beautiful design. It also does not replace product thinking.

## What external UI stacks solve

External stacks can improve:

- component implementation quality
- section-level polish and spacing rhythm
- typography pairing ideas
- interaction and motion pattern references
- responsive behavior patterns
- consistency in design system application
- premium visual execution support

They are a design execution layer, not a business strategy engine.

## What external UI stacks do not solve

External stacks cannot decide:

- business positioning
- offer architecture
- conversion target
- launch-kit output scope
- pricing logic
- trust strategy
- product promise
- site archetype

These decisions remain with AI Web Factory's internal engines.

## Core execution order

Future agents should use this order:

1. Business brief
2. Website Architecture + Prompt Composer
3. Full Site Builder Workflow
4. Launch Kit output plan
5. External UI Design Stack selection
6. Visual system adaptation
7. Implementation
8. Critique and learning loop

External support is intentionally placed after architecture and before implementation.

## Licensing and anti-copy policy (strict)

Before any external code or content reuse:

1. Verify license and attribution requirements.
2. Prefer principle extraction over direct copying.
3. Do not paste large external skill docs into this repository.
4. Do not copy branded assets, unique layouts, or recognizable motifs.
5. Attribute external sources when required.
6. If license is unclear, treat as inspiration only and do not copy.
7. Keep output original and business-specific.

Practical rule:

- if legality is uncertain, do not copy
- if resemblance is obvious, rework until original

## Selection model for external support

Use this decision table before selecting external support:

| Input factor | Recommended support mode | Typical stack emphasis | Main failure risk | Mitigation |
| --- | --- | --- | --- | --- |
| visual ambition level | low, medium, high | low: components; medium: components + UX rules; high: curated references + motion + components | style noise or underwhelming execution | align to conversion and trust goal first |
| site archetype | utility, productized, premium/campaign | utility: clarity components; productized: mixed; premium: selective high-craft patterns | mismatch with buyer expectations | map stack to archetype and trust problem |
| business category | local, SaaS/service, premium/creator, ecommerce | category-fit components and section jobs | generic cross-category sameness | apply Site DNA adaptation rules |
| motion/3D preference | none, subtle, expressive | none: static quality; subtle: micro-interactions; expressive: controlled moments | decorative motion overload | enforce motion intent and reduced-motion policy |
| implementation time budget | tight, moderate, extended | tight: proven primitives; moderate: adapted patterns; extended: bespoke systemization | overbuilding beyond scope | cap complexity by scope and conversion priority |
| components vs full visual language need | component-first, mixed, full-language | component stack vs multi-stack blend | fragmented visual identity | define shared visual system before sourcing |
| risk of over-design | low, medium, high | low risk allows stronger expression; high risk requires restraint | offer clarity loss | run anti-generic and anti-overdesign gates |
| accessibility/responsive needs | standard, strict | accessibility-first libraries and responsive-safe patterns | inaccessible or brittle components | require responsive/a11y checks before sign-off |

## Design execution stack types

### A. UI/UX skill/reference stack

Used for:

- taste rules
- style taxonomies
- UX heuristics
- palette logic
- typography systems
- layout principles

When to use:

- direction quality is weak or generic
- hierarchy and visual logic need reinforcement

When not to use:

- direction is already precise and validated

Expected output:

- explicit visual rules and guardrails tied to business-fit

Risks:

- abstract recommendations that never become implementation decisions

Integration checklist:

- convert references into explicit rules
- connect each rule to buyer trust and conversion logic
- reject purely trendy rules

### B. Component library stack

Used for:

- high-quality reusable sections and components
- navigation, pricing, forms, testimonials, accordions, carousels, bento layouts, proof modules

When to use:

- implementation speed and consistency are critical
- section jobs are clear and componentizable

When not to use:

- component family clashes with chosen visual system

Expected output:

- component shortlist mapped to section jobs

Risks:

- template assembly with weak originality
- style inconsistency from mixed component families

Integration checklist:

- map each component to conversion purpose
- adapt spacing, typography, states, and copy to business context
- remove decorative extras that do not improve conversion clarity

### C. Motion/interaction stack

Used for:

- subtle premium animation
- hover and reveal patterns
- selective 3D/motion moments when appropriate

When to use:

- visual hierarchy and perceived quality benefit from controlled motion

When not to use:

- trust-first utility pages where clarity and speed dominate

Expected output:

- motion policy and interaction behavior plan

Risks:

- overanimation
- performance/accessibility regressions

Integration checklist:

- define motion intent by section
- include reduced-motion behavior
- verify readability/performance impact

### D. Reference-site principle stack

Used for:

- extracting principles from best-in-class websites without copying

When to use:

- composition, rhythm, or proof sequencing needs stronger direction

When not to use:

- team cannot maintain strict anti-copy discipline

Expected output:

- principle list transformed into original structure/treatment

Risks:

- imitation drift
- recognizable layout borrowing

Integration checklist:

- transform each reference across story, structure, and visual treatment
- run anti-copy gates
- reject anything recognizable as a direct derivative

### E. Internal pattern memory stack

Used for:

- reusing lessons from AI Web Factory outputs
- carrying forward winning patterns
- rejecting known failure loops

When to use:

- similar conversion/trust problems have prior precedent

When not to use:

- old patterns conflict with new business context

Expected output:

- known-win patterns + known-fail rejection list for current scope

Risks:

- stale defaults becoming dogma

Integration checklist:

- validate relevance of prior lessons
- block repeated anti-pattern loops explicitly
- keep current business fit as final decision gate

## Visual direction adaptation to Site DNA

External style inputs must be adapted to Site DNA:

- business model
- target buyer
- trust problem
- conversion goal
- visual boldness level
- proof type
- industry texture

Rule:

- "looks cool" is not a valid reason for adoption
- every visual choice must improve comprehension, trust, or conversion

## Component sourcing workflow (required)

1. Define section jobs.
2. Identify component needs per section.
3. Select component families aligned with visual direction.
4. Adapt content and copy to the business.
5. Match all selected components to the shared visual system.
6. Validate responsiveness and accessibility.
7. Remove decorative complexity that does not improve outcomes.
8. Score clarity, trust, and polish.
9. Record adaptation and anti-copy outcomes.

## Anti-copy and anti-generic gates

Answer each with pass/fail and one evidence line:

- Is this a copied template?
- Does the page still fit the business model and buyer?
- Did external style overpower offer clarity?
- Is the visual anchor business-specific?
- Are components conversion-serving rather than trend-serving?
- Is responsive and accessibility quality acceptable?
- Would the page still work without the reference?
- Is the result original enough?

Fail on any gate means revise before claiming completion.

## Quality upgrade targets

The external stack layer should improve:

- first impression
- visual memorability
- polish and craftsmanship
- component quality
- spacing rhythm
- responsive behavior
- interaction feel
- conversion clarity
- perceived trust

It should not produce:

- style without strategy
- overanimated noise
- trendy but unclear interfaces
- inaccessible component choices
- generic template reuse
- irrelevant 3D/motion
- visual sameness across different business categories

## Example scenarios (8)

### 1) AI Web Factory Front Door v2

- chosen stack type: UI/UX reference + component library + internal pattern memory
- why: needs strong visual execution while preserving launch-kit-first product logic
- borrow as principles: conversion-first hero hierarchy, output-proof sequencing, high-quality intake modules
- avoid: dashboard-first hero, generic "AI platform" section stack, trend-only effects
- component candidates: intake preview modules, output artifact cards, plan/pricing sections, trust and FAQ rails
- visual adaptation notes: emphasize intake -> outputs -> trust -> start path
- anti-generic warning: reject if the page could belong to any generic SaaS builder

### 2) Real estate listing launch system

- chosen stack type: component library + reference-site principle stack
- why: listing workflows need operational clarity plus category trust signaling
- borrow as principles: process timelines, package comparisons, channel output previews
- avoid: generic automation copy and abstract metaphors unrelated to listings
- component candidates: workflow steps, proof gallery, package matrix, brokerage trust strip, demo CTA
- visual adaptation notes: anchor on listing-quality outcomes and compliance confidence
- anti-generic warning: reject if category specificity is not visible above fold

### 3) Local electrician/plumber site

- chosen stack type: component library stack
- why: immediate trust and contact conversion matter more than visual experimentation
- borrow as principles: first-screen action clarity, trust badges, service/FAQ compactness
- avoid: motion-heavy storytelling and irrelevant premium theatrics
- component candidates: sticky call/quote CTA, service area block, reviews, FAQ accordion, emergency-availability strip
- visual adaptation notes: prioritize "call now / request quote" confidence path
- anti-generic warning: reject if contact path is hidden or delayed

### 4) Premium spa/wellness studio

- chosen stack type: UI/UX reference + motion/interaction stack
- why: premium perception depends on controlled atmosphere and tactile polish
- borrow as principles: refined typography hierarchy, restrained motion, transformation-led proof pacing
- avoid: empty minimalism and overdecorated luxury cliches
- component candidates: signature offering modules, proof stories, booking CTA bands, trust/FAQ blocks
- visual adaptation notes: keep elegant tone while preserving booking clarity
- anti-generic warning: reject if style quality rises but conversion clarity drops

### 5) Creator course launch

- chosen stack type: UI/UX reference + component library
- why: needs momentum narrative and concrete curriculum/proof modules
- borrow as principles: transformation progression, objection-handling sequence, CTA urgency patterns
- avoid: hype without structure and novelty visuals without trust
- component candidates: curriculum grid, testimonial/result modules, pricing plans, FAQ, enrollment CTA bars
- visual adaptation notes: align each section to learner confidence progression
- anti-generic warning: reject if excitement is high but proof density is weak

### 6) Ecommerce skincare product

- chosen stack type: component library + reference-site principle stack
- why: purchase conversion depends on evidence, comparison clarity, and product trust
- borrow as principles: product-led composition, benefit-evidence sequencing, review prominence
- avoid: software-like feature walls and unrelated trendy effects
- component candidates: product hero media, ingredient/spec sections, comparison table, review modules, buy CTA
- visual adaptation notes: lead with product trust before checkout actions
- anti-generic warning: reject if abstract style replaces tangible product proof

### 7) Boutique agency/studio

- chosen stack type: UI/UX reference + reference-site principles + internal pattern memory
- why: needs distinct positioning and case-led credibility beyond visual novelty
- borrow as principles: case narrative composition, process framing, typographic contrast
- avoid: portfolio-wall sameness and agency cliche section flow
- component candidates: case cards, process timeline, service framing blocks, fit-based inquiry CTA
- visual adaptation notes: make business outcomes the core visual proof
- anti-generic warning: reject if the site is aesthetic-first but qualification-weak

### 8) Mobile app launch

- chosen stack type: component library + motion/interaction stack
- why: install conversion benefits from app-in-use presentation and polished transitions
- borrow as principles: behavior-first previews, use-case sequencing, proof-before-install flow
- avoid: desktop SaaS layout defaults and decorative motion without product meaning
- component candidates: app preview hero, use-case modules, social proof strip, app store CTA bands
- visual adaptation notes: keep mobile-first hierarchy and install intent dominant
- anti-generic warning: reject if app value is unclear in the first interaction

## Critique learning integration

When feedback appears, convert it into:

- external stack adjustment
- visual rule learned
- component/pattern rule learned
- future checklist gate

| Feedback | External stack adjustment | Visual rule learned | Component/pattern rule learned | Future gate |
| --- | --- | --- | --- | --- |
| "This looks too basic" | increase UI/UX reference support | strengthen hierarchy and anchor definition | replace weak default modules | does first read feel premium and specific? |
| "This is empty" | rebalance with component stack and proof modules | maintain meaningful visual weight | add conversion-bearing sections | does each section carry value, proof, and action? |
| "This looks like a wireframe" | increase reference-principle support | move from schematic layout to persuasive composition | replace placeholder modules | is this sell-ready or only structure-ready? |
| "This feels like generic SaaS" | tighten Site DNA adaptation and stack constraints | enforce category-specific visual cues | remove generic template sections | can this be confused with unrelated SaaS? |
| "This does not have wow factor" | add controlled motion/reference depth | increase memorability with purpose | upgrade hero/proof staging | is there one memorable business-fit anchor? |
| "This does not fit the business" | reduce external style influence and re-anchor | prioritize trust problem and buyer context | replace mismatched modules | does each major section fit this business model? |
| "This looks copied" | pull back to principle-only extraction | increase transformation distance from source | replace recognizable structures | would source resemblance still be obvious? |
| "This is overdesigned" | reduce motion and decorative component load | restore clarity-first rhythm | remove non-conversion visuals | does each visual choice improve comprehension or conversion? |

## How this changes future Front Door v2 work

For future AI Web Factory Front Door v2 efforts:

- agents should stop reinventing every UI block from scratch
- external component and pattern stacks can be used as optional execution accelerators
- architecture and launch-kit logic remain the controlling layer
- anti-copy and licensing checks become explicit pre-build gates
- critique outcomes directly refine future stack selection and adaptation rules

Expected impact:

- stronger high-fidelity execution quality
- less generic output
- lower copying/legal risk
- better alignment between premium visuals and product strategy

Again: this is an integration framework, not a guarantee of visual excellence.
